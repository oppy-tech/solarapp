## 2. Decisions We Rejected

### Single-agent approach
We considered having one agent do all three tasks sequentially. Rejected because:
- It doesn't demonstrate multi-agent orchestration (the core of Option 3)
- No parallelism — slower execution
- A single agent with broad file access has no guardrails against breaking its own earlier work

### Agents without a shared contract
Early on, the backend and frontend agents had no explicit data contract. This risked the backend passing variables the frontend didn't expect (or vice versa). We added a typed contract to both agent prompts so they can work independently without coordination overhead.

### Code-first (no TDD)
The initial agent prompts focused only on implementation. We added TDD requirements after recognising that without tests, there's no automated way to verify the agents' output — especially important when agents work in parallel and can't manually check each other.

### Agents without acceptance criteria
The technical requirements were clear, but they described *what to build*, not *what done looks like from a user's perspective*. We added a Product Owner agent to define acceptance criteria that the coding agents' tests map to. This closes the gap between "code works" and "feature is correct".

## 3. What Surprised Us

- **`setup.sh` assumed Sanctum was bundled with Laravel** — it isn't in Laravel 11. The stub `User.php` used `HasApiTokens` which immediately crashed migrations. This was the first thing that broke and set the tone: always verify the baseline before building on it.

- **The CloudFormation template has significant security issues** (hardcoded passwords, wildcard IAM, no WAF) that directly informed the guardrails we gave each agent. The infra context wasn't just background reading — it shaped concrete rules like "use parameterised queries" and "validate all input" with real justification.

- **The test database didn't exist** — PHPUnit was configured to use a `testing` database but Sail's MySQL only creates `laravel` by default. Had to manually `CREATE DATABASE testing` before tests could run with `RefreshDatabase`.

- **GitHub Actions required 7 iterations to get right** — `claude-code-action` needed `mode: agent` (not the default `tag`), `id-token: write` for OIDC auth, `direct_prompt` (not `prompt`), and the workflow file had to match the default branch exactly. Each agent also produced an `output.txt` log file that caused merge conflicts. The CI debugging loop was the most time-consuming part of the project.

- **Cross-agent contract mismatch was the hardest bug** — the frontend agent wrote tests against the pre-implementation controller (which hardcoded `'N/A'`). When the backend agent implemented the real calculation, the frontend test expected `N/A` where it should have expected `3 days, 6 hours`. This is the classic integration problem: agents working in parallel can't see each other's implementations.

- **PHP's `floor()` returns a float** — `floor(0) === 0` is `false` in PHP strict mode because `floor()` returns `0.0` (float), not `0` (int). The Blade template used `$days === 0` to decide whether to show minutes, which silently failed. A subtle type coercion bug that only manifests with strict comparisons.

## 4. Code We Don't Fully Understand

- **The backend agent's approval time calculation** uses `diffInSeconds()` in a PHP loop rather than a database-level `AVG(TIMESTAMPDIFF(...))`. The agent commented this was for "cross-database compatibility" (SQLite vs MySQL). This works but loads approved projects into memory — acceptable for 160 records but wouldn't scale to thousands without refactoring to a raw SQL aggregate.

- **The data integrity agent's Artisan command** generates a full markdown report in PHP string concatenation. The structure is correct but the report generation logic is tightly coupled to the output format — changing the report structure would require editing the command internals. A template-based approach would be cleaner but wasn't required for the scope.

- **`peter-evans/create-pull-request` behaviour** — this action creates commits and opens PRs, but when the target branch already has the content pushed by a prior job, it sees no diff and silently does nothing. We worked around this by creating the PR via `gh` CLI directly.

## 6. Multi-Agent Workflow (Option 3 Specific)

### How we decomposed the work
Four agents, each with a single responsibility:

| Agent | Responsibility | Owns | Tests |
|-------|---------------|------|-------|
| Product Owner | Acceptance criteria, validation | `VALIDATION_REPORT.md` | N/A (validates others) |
| Backend | Controller logic (filtering, stats, pagination) | `DashboardController.php` | `DashboardControllerTest.php` |
| Frontend | Blade template (UI, display, navigation) | `dashboard.blade.php` | `DashboardViewTest.php` |
| Data Integrity | Audit command and report | `DataIntegrityCheck.php`, `DATA_INTEGRITY_REPORT.md` | `DataIntegrityCheckTest.php` |

### Guardrails we set up

1. **File ownership** — each agent can only edit its designated files. This prevents merge conflicts and accidental breakage.
2. **Data contract** — backend and frontend share a typed contract (what variables the controller passes to the view). They can work in parallel without coordination.
3. **TDD** — every agent writes tests first. Tests are the verification mechanism — if they pass, the AC is met.
4. **AC mapping** — each agent's tests explicitly reference acceptance criteria from the PO agent. No test exists without a reason.
5. **Infrastructure-informed security** — guardrails aren't generic best practices; they're derived from actual issues in the production CloudFormation template (e.g. "validate input because there's no WAF").
6. **Query efficiency** — explicit rules against `->get()` on unbounded queries, with specific alternatives (DB aggregates, `->paginate()`, `->chunk()`), informed by the `db.t4g.small` constraint.
7. **Multi-tenancy** — every agent that touches data is warned about the 2-AHJ setup and told to scope through `$ahj->projects()`.

### Where agents succeeded and where they failed

**Successes:**
- **Backend agent** — 14 tests, all passing on first merge. Clean implementation with date filtering, approval time calculation, pagination, and proper multi-tenancy scoping. Followed the data contract exactly.
- **Data integrity agent** — 18 tests, all passing. Produced a comprehensive `DATA_INTEGRITY_REPORT.md` with findings across both AHJs. Correctly identified status/timestamp inconsistencies, missing fields, and impossible data.
- **Frontend agent** — 11 tests, clean Blade implementation with date picker, approval time display, pagination. Good Tailwind styling consistent with existing patterns.
- **File ownership** — zero merge conflicts in application code across all three agents. The branch-per-agent strategy worked perfectly.
- **TDD approach** — every agent wrote tests first, giving us 37 automated checks across the entire feature set.

**Failures:**
- **Frontend agent's contract test** — the frontend agent wrote `assertSee('N/A')` for a test that creates an approved project with valid timestamps. It tested against the pre-implementation controller (which hardcoded `'N/A'`) rather than the post-implementation behaviour. This is the key limitation of parallel agents with a shared contract: the frontend couldn't see the backend's implementation.
- **PHP type coercion bug (first run)** — the frontend agent used `floor()` with strict `===` comparison, which fails because `floor()` returns float. The second agent run used loose `==` which worked, but this is a subtle PHP gotcha that the agent prompt should have warned about.
- **`submitted_at` null handling (first run)** — the Blade template called `->format()` on a nullable `submitted_at` without null-safe operator. The second run handled this correctly with `?->`.

### What we'd do differently next time

1. **Add a contract validation step** between agent execution and merge — a lightweight check that the backend's return types match what the frontend expects, catching mismatches before tests run.
2. **Include PHP-specific gotchas in the frontend prompt** — `floor()` returns float, nullable column access needs `?->`, `===` vs `==` for type safety.
3. **Run agents sequentially with a dependency graph** — backend first, then frontend (so it can test against the real implementation), then data integrity (independent). The parallelism saved ~5 minutes but cost us a cross-agent bug.
4. **Bake the CI fixes into the workflow from the start** — `mode: agent`, `id-token: write`, `direct_prompt`, `.env.example` setup, `output.txt` conflict handling. These are learnable patterns for `claude-code-action` that should be documented upfront.
5. **Use `gh pr create` instead of `peter-evans/create-pull-request`** — simpler, more predictable, and avoids the "no diff" edge case when branches are pushed by prior jobs.
