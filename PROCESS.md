## 2. Decisions We Rejected

### Single-agent approach
We considered having one agent do all three tasks sequentially. Rejected because:
- Doesn't demonstrate multi-agent orchestration (the core of Option 3)
- No parallelism — slower execution
- A single agent with broad file access has no guardrails against breaking its own earlier work

### Agents without a shared data contract
Early on, the backend and frontend agents had no explicit contract. This risked the backend passing variables the frontend didn't expect. We added a typed contract (`stats`, `projects`, `filters`) to both agent prompts so they can work independently. This proved critical — when the contract was implicit, the frontend tested against the pre-implementation controller and got wrong assertions.

### Code-first (no TDD)
The initial agent prompts focused only on implementation. We added TDD requirements after recognising that without tests, there's no automated way to verify the agents' output — especially important when agents work in parallel and can't manually check each other.

### Agents without acceptance criteria
The technical requirements described *what to build*, not *what done looks like*. We added a Product Owner agent to define acceptance criteria. This closed the gap between "code works" and "feature is correct", and gave us a structured framework for the code review agent.

### TDD without a green gate
The first iteration told agents to "follow TDD" but didn't enforce that all tests pass before committing. Agents interpreted this as "write tests, then implement" — but stopped before all tests were green. We added two layers of enforcement:
1. **Prompt-level**: "CRITICAL: you must not finish until ALL tests pass"
2. **CI-level**: a `Verify all tests pass` step that runs the test suite independently and blocks the push

### `peter-evans/create-pull-request` for PR creation
This GitHub Action creates commits and PRs, but when the merge job already pushes the branch, the action sees no diff and silently does nothing. We replaced it with `gh pr create` which is simpler and more predictable.

### Happy-path-only testing
The first round of tests only covered happy paths. The code review flagged issues (SQL injection, XSS, null crashes) that negative tests would have caught. We added comprehensive negative test requirements: SQL injection, XSS payloads, invalid dates, boundary conditions, null fields, extreme pagination, special characters.

## 3. What Surprised Us

- **`setup.sh` assumed Sanctum was bundled with Laravel** — it isn't in Laravel 11. The stub `User.php` used `HasApiTokens` which immediately crashed migrations. Set the tone: always verify the baseline before building on it.

- **The CloudFormation template has significant security issues** (hardcoded passwords, wildcard IAM, no WAF) that directly informed the guardrails we gave each agent. The infra context wasn't just background reading — it shaped concrete rules like "use parameterised queries" and "validate all input" with real justification from the production architecture.

- **GitHub Actions required 7+ iterations** — `claude-code-action` needed `mode: agent`, `id-token: write`, `direct_prompt`, and the workflow file had to match the default branch. Each agent produced an `output.txt` log that caused merge conflicts. The `always()` condition propagated skip states to downstream jobs. The testing DB needed explicit credentials in `.env`. The CI debugging loop was the most time-consuming part of the project.

- **Cross-agent contract mismatch was the hardest bug** — the frontend agent wrote tests against the pre-implementation controller (which hardcoded `'N/A'`). When the backend agent implemented the real calculation, the frontend test expected `N/A` where it should have expected `3 days, 6 hours`. This is the fundamental tension of parallel agents: speed vs integration correctness.

- **PHP's `floor()` returns a float** — `floor(0) === 0` is `false` in PHP strict mode. The Blade template used `$days === 0` to decide whether to show minutes, which silently failed. A subtle type coercion bug that only manifests with strict comparisons.

- **Agents will commit with failing tests if you let them** — telling an agent to "follow TDD" is not the same as telling it "do not stop until 0 failures." The first is a process instruction; the second is an exit condition. Without the explicit exit condition and a CI gate, agents treated "wrote tests and implemented" as done, even with 25 failures.

- **Known issues from code review feed forward effectively** — adding a "Known Issues (MUST FIX)" section to agent prompts with specific bugs from the prior run gave agents clear, actionable context. This creates a feedback loop: code review → prompt update → next run fixes the issues. This was more effective than generic guardrails.

## 4. Code We Don't Fully Understand

- **Cross-database compatibility for approval time** — the backend agent may use `diffInSeconds()` in a PHP loop rather than `TIMESTAMPDIFF()` for SQLite/MySQL compatibility. The trade-off between DB portability and query efficiency is a judgment call; the agent prompt now asks for DB aggregates but the agent may still choose the PHP approach and document why.

- **Data integrity command's report generation** — generates markdown via PHP string concatenation. Correct but tightly coupled to the output format. A Blade template or dedicated formatter would be cleaner but wasn't required.

- **`claude-code-action` OIDC token flow** — the action fetches a GitHub OIDC token to authenticate. The exact handshake between `id-token: write`, the workflow file match requirement, and the `mode: agent` setting is opaque. We discovered the requirements through error messages, not documentation.

## 6. Multi-Agent Workflow (Option 3 Specific)

### How we decomposed the work

Four agents, each with a single responsibility and strict file ownership:

| Agent | Responsibility | Writes | Tests | ACs |
|-------|---------------|--------|-------|-----|
| Product Owner | Acceptance criteria (34 ACs across 7 categories) | — | — | Defines all |
| Backend | Controller logic (filtering, stats, pagination, validation) | `DashboardController.php` | `DashboardControllerTest.php` | AC1.2–1.8, AC2.2–2.4, AC3.1, AC3.4, AC5.1–5.3, AC7.1–7.9 |
| Frontend | Blade template (UI, display, navigation, escaping) | `dashboard.blade.php` | `DashboardViewTest.php` | AC1.1, AC1.4–1.6, AC2.1, AC2.4, AC3.2–3.3, AC3.6, AC6.2–6.3, AC7.3, AC7.10 |
| Data Integrity | Audit command and report | `DataIntegrityCheck.php`, `DATA_INTEGRITY_REPORT.md` | `DataIntegrityCheckTest.php` | AC4.1–4.8 |

### Guardrails we set up

1. **File ownership** — each agent edits only its designated files. Enforced by prompt instruction and verified by code review agent. Result: zero merge conflicts in application code across all runs.

2. **Data contract** — typed interface between backend and frontend documented in both prompts. The controller returns a specific shape (`stats`, `projects`, `filters`); the view consumes it. Agents can work in parallel without coordination.

3. **TDD with green gate** — two layers of enforcement:
   - Prompt: "CRITICAL: you must not finish until ALL tests pass. Debug, fix, re-run."
   - CI: `Verify all tests pass` step runs the test suite independently and blocks the push.

4. **AC mapping** — every test maps to a specific acceptance criterion. No test exists without a reason. The PO agent defines 34 ACs across 7 categories; each coding agent's prompt lists which ACs it owns.

5. **Infrastructure-informed security** — guardrails derived from real issues in `solarapp.yaml`:
   - Hardcoded DB password → "never hardcode credentials, use `env()`"
   - Wildcard IAM (`s3:*`, `sqs:*`) → "be explicit about services used"
   - Port 80 open to `0.0.0.0/0` with no WAF → "all input MUST be validated"
   - Single `db.t4g.small` → "use DB aggregates, not PHP loops"

6. **Query efficiency** — explicit rules against `->get()` on unbounded queries:
   - Stats: `->count()`, `->avg()` at query builder level
   - Pagination: `->paginate(20)`, never get-then-slice
   - Filtering: `->whereBetween()` at query level, not `->get()->filter()`
   - Batch processing: `->chunk()` or `->cursor()` for full-table scans
   - N+1 prevention: `->with()` for eager loading

7. **Multi-tenancy** — every agent that touches data is warned about the 2-AHJ setup and told to scope through `$ahj->projects()`. Tested with projects across both AHJs.

8. **Negative testing** — comprehensive edge case coverage required:
   - Backend: SQL injection, XSS, invalid dates, end-before-start, empty params, partial params, extreme pagination, boundary dates, zero-duration approval
   - Frontend: XSS escaping in params and table output, null field rendering, special characters, time format edge cases (hours-only, minutes-only, near-zero, very large)
   - Data integrity: null/empty fields, orphaned references, impossible timestamps, future dates, unreasonably fast approvals, severity separation, exit codes

9. **Known issues feed-forward** — agent prompts include a "Known Issues from Code Review (MUST FIX)" section listing specific bugs from prior runs. This creates a feedback loop: code review → prompt update → next run fixes the issues.

### Where agents succeeded

- **File ownership worked perfectly** — zero merge conflicts in application code across all runs. The branch-per-agent strategy with strict file boundaries is the strongest guardrail.
- **TDD produced high coverage** — agents wrote 60+ tests covering happy paths, negative cases, and edge cases. The test suite catches real bugs (null crashes, type coercion, missing stat cards).
- **Data integrity agent was the most self-contained** — owned its own command, tests, and output. No cross-agent dependencies. Produced a comprehensive report with actionable findings.
- **Code review agent provided actionable feedback** — identified performance violations (DB aggregate vs PHP loop), security gaps (null-safe operator), and missing functionality (pending stat card). Posted structured review on the PR with AC coverage table.

### Where agents failed

- **Cross-agent contract mismatch** — the frontend agent tested against the pre-implementation controller. Parallel execution means agents can't see each other's work. The data contract mitigated this but didn't eliminate it for test assertions.
- **Agents committed with failing tests** — without an explicit "0 failures" exit condition and a CI gate, agents treated "wrote tests and implemented" as done. Required prompt-level and CI-level enforcement.
- **Backend agent violated its own guardrails** — used `->get()` + PHP loop for approval time despite the prompt saying to use DB aggregates. The agent chose "cross-database compatibility" over performance. The known issues feed-forward addresses this.
- **Data integrity agent's fast-approval check was buggy** — `diffInSeconds()` produced negative values when `approved_at < submitted_at`, causing 68 misleading warnings. The known issues feed-forward addresses this.
- **Negative tests exposed implementation gaps** — agents wrote the negative tests (good) but didn't always implement the handling (bad). 25 of 69 tests failed on the third run. The green gate now blocks this.

### What we'd do differently next time

1. **Enforce green tests from the start** — the "CRITICAL: 0 failures" prompt instruction and CI gate should have been in place from the first run, not added after agents committed with failing tests.

2. **Add a contract validation step** — automatically verify that the backend's return types match what the frontend expects before the merge, catching mismatches without running the full test suite.

3. **Include language-specific gotchas in prompts** — PHP's `floor()` returns float, nullable columns need `?->`, `===` vs `==` matters. These are predictable pitfalls that should be documented upfront.

4. **Consider sequential execution for dependent agents** — backend first, then frontend (so it tests against the real implementation). The parallelism saves time but costs integration correctness. Data integrity can always run in parallel.

5. **Bake CI patterns into the workflow from the start** — `mode: agent`, `id-token: write`, `direct_prompt`, `.env.example` setup, `output.txt` conflict handling, `gh pr create` over `peter-evans`. These are learnable patterns for `claude-code-action` that cost us 7+ iterations.

6. **Use the known issues feed-forward pattern from the start** — don't wait for a code review to discover issues. Proactively list common failure modes in the agent prompts based on the tech stack (Laravel, Blade, Eloquent, PHP type system).
